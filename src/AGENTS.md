# Source Domain AGENTS

## 제품 원칙
- ParrotKit은 단순히 영상을 요약하는 툴이 아니라, 레퍼런스 분석을 실행 가능한 콘텐츠 레시피로 바꾸는 툴이다.
- 제품의 진실값은 `원본 분석 -> 레시피 -> 촬영 실행`의 연속된 흐름에 있다.
- 사용자에게는 하나의 제품으로 보여야 하며, 분석 툴 / 레시피 툴 / 촬영 툴로 쪼개진 경험을 만들지 않는다.

## 원본 분석과 레시피의 역할
- 원본 분석은 `왜 이 영상이 먹히는가`를 설명하는 근거 레이어다.
- 레시피는 `내가 무엇을 말하고, 어떻게 보여주고, 어떤 순서로 재구성할 것인가`를 담는 실행 레이어다.
- 레시피만 있고 분석이 없으면 결과물은 생기지만 학습과 납득이 약해진다.
- 앞으로 recipe 관련 작업은 `원본 분석`과 `실행 레시피`를 의도적으로 구분해 설계한다.

## 입력 구조
- 레퍼런스 영상 URL은 분석 대상이다.
- 기업 가이드라인 PDF, 제품 소개서, 브리프 문서는 브랜드 컨텍스트다.
- 레퍼런스와 브랜드 컨텍스트는 같은 입력으로 취급하지 않는다.
- PDF는 raw text로 계속 재주입하기보다, 먼저 구조화된 brief JSON으로 정규화한 뒤 재사용한다.

## 브랜드 컨텍스트 추출 기준
- PDF/브리프에서 우선 추출할 항목:
  - 브랜드명
  - 제품명
  - 캠페인 목적
  - 타겟 고객
  - 핵심 소구점
  - 보조 근거와 증빙 포인트
  - 필수 포함 문구
  - 금지 표현
  - 필수 태그/광고 고지
  - 촬영 가이드
  - 편집 가이드
  - CTA 방향
- 이후 recipe 생성은 이 structured brief를 source of truth로 사용한다.

## 출력 구조
- 결과 화면은 역할상 최소한 아래 세 층을 염두에 둔다.
  - `Analysis`: 원본이 왜 작동하는지
  - `Recipe`: 컷별 실행 지시와 재구성 흐름
  - `Shooting` 또는 `Prompter`: 실제 촬영 중 보는 화면
- `Recipe`는 보기 좋은 요약이 아니라, 실제로 다시 찍을 수 있는 수준의 지시여야 한다.

## 프롬프터 원칙
- 프롬프터는 촬영 실행 도구다.
- 카메라 위에는 사용자가 선택한 정보만 띄울 수 있어야 한다.
- 후보 정보 예시:
  - 대사
  - 키워드
  - 컷의 핵심 소구점
  - 필수 포함 문구
  - 금지 표현 경고
  - CTA
- 프롬프터 요소는 위치 조정이 가능해야 하고, 장기적으로는 drag-and-drop 편집을 지원하는 방향으로 본다.
- 프롬프터는 읽기 쉽게 큰 타이포와 명확한 안전영역을 우선한다.

## 카메라와 네이티브 방향
- 촬영 experience는 추후 네이티브 surface로 강화할 수 있다.
- 다만 제품을 둘로 나누지 않는다. 레시피/분석은 같은 제품 안에서 유지한다.
- 네이티브로 확장하더라도 핵심 개념은 `앱 내부 카메라 프리뷰 + 프롬프터 오버레이`다.
- 시스템 카메라 앱으로 넘기는 handoff 방식은 프롬프터 경험을 약하게 만들 수 있으므로 신중히 본다.
- 현실적인 방향은 `대부분의 product flow는 웹`, `shooting surface만 네이티브 강화`다.

## 구현 판단 기준
- 새 기능이 들어올 때마다 먼저 묻는다:
  - 이 기능은 원본 분석 레이어인가?
  - 실행 레시피 레이어인가?
  - 브랜드 컨텍스트 레이어인가?
  - 실제 촬영 실행 레이어인가?
- 각 레이어의 역할이 섞이면 UX와 데이터 구조를 다시 분리한다.
- 제품 카피와 UI는 항상 `분석 도구`보다 `촬영 가능한 실행 도구` 쪽을 강화하는 방향으로 판단한다.
